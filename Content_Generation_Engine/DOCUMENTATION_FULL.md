### 1) Executive summary

The Content Generation Engine is a compact pipeline designed to produce SEO-optimized blog posts for a sponsoring company (Reticulo Sports Pvt Ltd) using two main inputs:

- Trending keywords (sourced from Google Trends via `pytrends`), and
- Company metadata (provided in `company_config.json`).

The pipeline runs the keywords through a lightweight trend analysis prompt, and then builds a blog generation prompt which it sends to the LLM via `groq_client.ask_groq`. The raw model output (Markdown or text) is converted into HTML and injected into an HTML template, producing complete standalone blog HTML files under `generated_blogs/` following a consistent naming scheme.

An important design decision: the system accepts any company metadata (e.g., Netflix, Nike) and will generate a piece of content targeted at that company. However, the engine is configured to connect every generated blog back to the sponsoring company (Reticulo Sports Pvt Ltd) — i.e., the blog will include a narrative tying the product/company used as input to benefits, opportunities, or services that Reticulo Sports offers. This is controlled by the prompts and the `company_info` object passed into `generate_blog`.

### 3) Detailed module walkthrough

Pipeline / Full Data Flow:
1. Run `main.py` ([main.py](main.py#L1)).
2. `trends_fetcher.get_trending_keywords()` obtains trending queries via Google Trends (`pytrends`). If Google Trends fails or rate-limits, a fallback set of keywords is returned.
3. `trend_analyzer.analyze_trends(keywords, company_info)` calls `groq_client.ask_groq` with `TREND_ANALYSIS_SYSTEM_PROMPT` (from `prompts.py`) to get a short analysis and content opportunities.
4. `blog_generator.generate_blog(keywords, company_info)` builds a user prompt including company metadata and the top keywords, then calls `groq_client.ask_groq` with `BLOG_GENERATION_SYSTEM_PROMPT` to obtain the blog text.
5. If the returned content is Markdown, it is converted to HTML with the `markdown` package. If conversion fails, a simple fallback replacement is used.
6. The HTML is injected into `templates/blog_template.html` replacing `{{BLOG_CONTENT}}` and saved under `generated_blogs/` using the naming pattern: `{slug}_blog_{YYYYMMDD_HHMMSS}.html`.


All code files are small and self-contained — this section explains each one.

#### 3.1 `main.py`

- Responsibilities:
  - Orchestrate the pipeline: fetch trends, run analysis, generate blog
  - Load `company_config.json`
- Key functions/logic:
  - `load_company_config(path=CONFIG_PATH)`:
    - Validates file existence, opens the JSON, and returns a Python dict
  - `main()`:
    - Calls `get_trending_keywords()`
    - Calls `analyze_trends(keywords, company_info)` and prints analysis
    - Calls `generate_blog(keywords, company_info)` and prints the saved blog path

Notes/caveats:
  - `main.py` uses synchronous, blocking calls — there is no concurrency or retries built-in.
  - Errors in loading `company_config.json` raise `FileNotFoundError`, which is not explicitly caught in `main()` so that failure is visible.

#### 3.2 `trends_fetcher.py`

- Purpose: obtain relevant trending queries. Implementation uses `pytrends`.
- Important details:
  - `TrendReq(hl="en-US", tz=330)`: This sets locale and timezone; tz=330 corresponds to IST (UTC+5:30). The timezone affects how Google Trends interprets daily windows.
  - `build_payload(fallback_keywords, timeframe="now 7-d")`: The code uses a fallback set of base keywords to build a query. `timeframe="now 7-d"` restricts trends to the last 7 days.
  - `related_queries()`: For each base keyword, the function accesses the `.get("top")` DataFrame and takes the `query` column head(5) — up to 5 top related queries per keyword.
  - `results = list(dict.fromkeys(results))` preserves insertion order while de-duplicating.
  - Returns up to 15 deduped queries.

Failure handling:
  - `exceptions.TooManyRequestsError`: prints a 429 warning and returns fallback keywords.
  - Generic `Exception` catches other errors and returns fallback keywords.

Potential improvements:
  - Add exponential backoff and retries on 429
  - Cache results to avoid repeated calls during development or CI
  - Accept a `region` and `timeframe` as parameters for flexible usage

#### 3.3 `trend_analyzer.py`

- Purpose: produce an analysis of trending keywords, optionally tailored to a company.
- Behavior:
  - Combines keywords into `keyword_text` and uses `TREND_ANALYSIS_SYSTEM_PROMPT` (from `prompts.py`) as the system prompt.
  - If `company_info` provided, adds `Company:` and `Industry:` lines to the user prompt to give context.
- Returns: the raw model response string from `ask_groq`.

How the output is intended to be used:
  - The returned analysis is printed to the console in `main.py`.
  - It can be used to refine the blog prompt or to provide a summary for editors.

#### 3.4 `blog_generator.py`

- Responsibilities:
  - Build a blog prompt containing `company_info` and `keywords`
  - Send prompt to the model via `ask_groq` (with `BLOG_GENERATION_SYSTEM_PROMPT`)
  - Convert Markdown to HTML (using `markdown` package)
  - Inject into `templates/blog_template.html` and write to `generated_blogs/`

Key functions and their logic:
  - `_slugify(value)`
    - Implementation: returns a string with lowercase alphanumerics preserved and non-alphanumerics replaced by underscores. Leading/trailing underscores stripped.
    - Use: produces a filesystem-friendly slug for the company name.
  - `generate_blog(keywords, company_info)`
    - Creates `OUTPUT_DIR` via `os.makedirs(OUTPUT_DIR, exist_ok=True)`
    - Pulls company metadata fields with safe defaults: `company_name`, `industry`, `target_audience`, `services`.
    - Builds `user_prompt` string with these fields and the `keyword_text`.
    - Calls `ask_groq(BLOG_GENERATION_SYSTEM_PROMPT, user_prompt)` to get blog text.
    - Converts returned string to HTML using: `blog_html = markdown(blog)` inside try/except. Fallback replaces `**` with `<b>` and `\n` with `<br>`.
    - Reads `templates/blog_template.html` and replaces `{{BLOG_CONTENT}}` with `blog_html`.
    - Creates `filename` using `_slugify(company_name)` and timestamp format `%Y%m%d_%H%M%S`.
    - Writes the file as UTF-8 and returns the filepath.

Notes and edge cases:
  - If the model returns raw HTML (instead of Markdown), `markdown()` will still wrap or escape some constructs; the code does not explicitly detect or sanitize raw HTML.
  - The fallback conversion `blog.replace("**", "<b>").replace("\n", "<br>")` is simplistic: it produces unbalanced `<b>` tags and does not escape HTML entities.

#### 3.5 `groq_client.py`

- Purpose: abstraction layer over the Groq API for chat completions.
- Behavior:
  - Loads `.env` using `dotenv.load_dotenv()` and reads `GROQ_API_KEY`.

  - On `KeyboardInterrupt` or other `Exception` returns either a short fallback HTML string or a long fallback Markdown string.

#### Prompt engineering tips and recommendations

1. Ask for output format explicitly. For example: "Return the blog in Markdown using headings (H1, H2), bulleted lists for tips, and include the keywords exactly as provided at least 2-3 times each." Explicit structure reduces the chance the model returns unexpected HTML or JSON.
2. Ask the model to return a small machine-readable metadata block (e.g., YAML frontmatter) that includes `title`, `meta_description`, and `keywords_used`. That would enable automated extraction of meta tags for SEO.
3. Control verbosity: add instructions like "Limit the blog to 700-1100 words".
4. For the Reticulo tie-in: provide an explicit instruction in the prompt to include a short section "How Reticulo Sports helps" that maps the input company's products to Reticulo services.

### 7) Failure modes, errors, and mitigations

Failure: Google Trends rate limits (HTTP 429)
Cause: Unthrottled `pytrends` requests.
Symptom: `TooManyRequestsError` caught in `trends_fetcher`.
Mitigation:
- Implement exponential backoff and retry.
- Cache responses in a local file or in Redis for N minutes.

Failure: Missing or invalid `GROQ_API_KEY`
Cause: `.env` undefined or wrong value.
Symptom: `ask_groq` raises exception, returns fallback content.
Mitigation:
- Validate environment at startup; log a clear error and exit or run in offline mode.
- Use a secrets manager (Azure Key Vault, AWS Secrets Manager) in production.

Failure: Model returns unexpected format (raw HTML, JSON, or extremely long output)
Cause: Lack of strict output format constraint in the prompt.
Mitigation:
- Update prompts to request a strict Markdown or JSON format.
- Post-validate the output: if it contains HTML tags at the top-level, treat as HTML; if it looks like Markdown, convert; otherwise sanitize.

Failure: XSS and injection risks when inserting model output into HTML template
Cause: Trusting model output without sanitization.
Mitigation:
 - Sanitize model output before embedding (e.g., `bleach` to whitelist tags), or keep only trusted Markdown conversions.
 - Serve content from static hosting with correct Content Security Policy (CSP).

Failure: Filesystem or encoding errors writing the HTML
Cause: Unexpected characters, missing directory permissions
Mitigation:
- Ensure `os.makedirs(..., exist_ok=True)` is used (already present).
- Use `encoding='utf-8'` when writing files (present).

11) Productionization & scaling recommendations
---------------------------------------------

If this project will be used in production or as a service, consider:

- Queue-based processing: Accept generation requests and push them to a queue (Redis/RabbitMQ) so a worker can process them asynchronously.
- Rate limit & throttle calls to external APIs: Add retry/backoff and circuit breakers.

- Deploy as a containerized worker with stateless execution. Example `Dockerfile`:

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
